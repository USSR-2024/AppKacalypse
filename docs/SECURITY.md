# SECURITY — AppKacalypse

Модель безопасности проще обычной: сервер один, наружу за обработкой не ходит.

## Принципы (ТЗ §10)
1. Все данные (задачи, календарь, токены) — только на этом сервере.
2. Внешние LLM-API не используются → пользовательский текст никуда не уходит.
3. Секреты — только в `.env` или Docker secrets. `.env` не коммитится; `.env.example` коммитится.
4. Webhook закрыты allowlist/token (Telegram `TELEGRAM_ALLOWED_USER_IDS`).
5. LLM не получает секреты, пароли, ключи, приватные конфиги.
6. Логи без API-ключей/токенов/паролей — regex-redaction в LLM Gateway как fail-safe.
7. Неоднозначные задачи — через confirmation flow.
8. Автоудаление/автоизменение задач без подтверждения запрещено.
9. Доступ к серверу: SSH-ключ, нестандартный порт 19949, пароль off, fail2ban.

## Доступ к серверу
- SSH только порт **19949** (22 закрыт), root `prohibit-password`, вход по ключу.
- fail2ban whitelist: owner `46.28.88.106` + control `89.125.2.39`.
- cloud-init отключён (затирал ключи) — `/etc/cloud/cloud-init.disabled`.

## Redaction (логи)
LLM Gateway маскирует перед записью в лог:
- логины вида `user@host` / `root@1.2.3.4` → `[SERVER_LOGIN]`
- API-ключи (`sk-...`, `Bearer ...`, длинные base64/hex) → `[API_KEY]`
- пароли в `password=...` → `[PASSWORD]`

Поскольку внешних LLM нет, утечки наружу не происходит — redaction нужен только для логов.

## Секреты в проде
- Google Calendar credentials — через Docker secret `/run/secrets/...`, не в `.env`.
- Бэкапы `.env` и ключей — вне Git, отдельным защищённым процессом.
