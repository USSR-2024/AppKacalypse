# CALENDAR — модель календаря

## Принцип
**Vikunja — единственная точка правды.** Календари пользователей — «витрина»:
синхронизация **односторонняя, Vikunja → календарь**. В чужие календари мы не пишем
и из них не читаем.

```
Telegram/email → LLM → Vikunja (источник правды)
                          │  ICS / CalDAV feed (read-only)
                          ▼
        Личные календари пользователей (Google / Apple / любой)
```

## Как это работает (универсально, «каждый сам»)
Vikunja из коробки отдаёт задачи как **CalDAV / ICS-фид**. Пользователь один раз
подписывается на свою персональную ссылку в любом календаре — и видит там все
задачи с дедлайнами. Ноль чужих паролей, ноль адаптеров, работает у всех.

- Ссылка-подписка хранится в `vikunja/seed/users.json` → поле `calendar_feed`.
- Google: «Другие календари → Подписаться по URL». Apple: «Новая подписка».
- Обновляется автоматически (клиент календаря периодически перечитывает фид).

## Событие vs задача в этой модели
Календарный intent (`create_calendar_event`, `create_task_with_calendar_event`)
остаётся в intent-модели, но в MVP **событие материализуется как задача Vikunja**
с датой/временем — а календарь подхватывает её через фид. Мы НЕ создаём объекты
напрямую в чужих календарях.

Правила (ТЗ §9.2) применяются к Vikunja:
- Только дедлайн → задача с due date.
- Встреча/созвон с датой+временем → задача со временем (в календаре видна как событие).
- Связанные «событие + доп. задача» → две связанные задачи Vikunja.

## Env
```
CALENDAR_PROVIDER=disabled   # в текущей модели не используется для записи;
                             # фид Vikunja работает независимо от этой переменной
```
Поля `GOOGLE_CALENDAR_*` / `CALDAV_*` в `.env.example` оставлены под будущий
push-режим (см. ниже), в MVP не задействованы.

## Точка расширения — двусторонняя синхронизация (FUTURE, не в MVP)
Если когда-нибудь понадобится «создал в личном календаре → появилось в Vikunja»
или «писать события прямо в личные календари людей»:
- добавить CalDAV-write адаптер (`services/calendar-adapter/`);
- per-user credentials (CalDAV URL + app-password) в справочнике;
- polling + дедуп + разрешение конфликтов.

**Сознательно отложено:** ломает принцип единой точки правды, дорого в поддержке,
а основной ввод и так идёт через Telegram→LLM→Vikunja.
